Get started

Request an account and begin building your integration

The NPGW API is organized around the principles of REST. It has predictable resource-oriented URLs, accepts and returns JSON-encoded bodies and responses, all while using standard HTTP response codes & authentication.

You can use the NPGW API via our Sandbox while completing your integration, which won’t affect your live data or interact with the Acquiring networks. The domain and credentials you use to authenticate and submit yours request determines whether a payment is processed via Production or Sandbox.

The NPGW API doesn’t currently support bulk updates. You can work on only one object per request.

Usage

There are multiple ways you can explore the Merchant API:

  • By using the classic Swagger toolset; view the definition using the Swagger Editor
  • By using Postman  to import the API definition and create your personal collection of requests.

Authentication

The NPGW API uses a Secret Token to authenticate requests. You can view and manage your tokens within your account.

Your Secret Token  carries many privileges, so be sure to keep it secure! Do not share your Token in publicly accessible areas such as GitHub, client-side code, and so forth.

All API requests must be made over HTTPS . Calls made over plain HTTP will fail. API requests without authentication will also fail.

Environments

The NPGW provides two separate domains for our API: one for Sandbox and one for Production. The sandbox environment is intended for testing and development, while the production environment is used for live transaction processing. All API endpoints are consistent across both environments, meaning the request structure, parameters, and responses are the same—only the base domain changes depending on the environment.

Successful acceptance testing is required in order to move your integration from Sandbox to Production.

SANDBOX BASE DOMAIN

https://sandbox.npgw.xyz

PRODUCTION BASE DOMAIN

https://npgw.xyz

Company Registration

Once you have been successfully set up as a Company within the NPGW, our team will configure Business Units  as required for payment processing under your organisational hierarchy.

Each Business Unit is assigned a unique merchantId and requires its own Secret Token to interface with the Merchant API.

You can retrieve these details by logging into your account and navigating to the System Administration menu.

Errors

The NPGW uses conventional HTTP response codes to indicate the success or failure of an API request. In general: Codes in the 2xx  range indicate success. Codes in the 4xx  range indicate an error that failed given the information provided (e.g. a required parameter was omitted etc). Codes in the 5xx  range indicate an error with our servers (these are rare).

API-level errors (such as invalid request parameters, authentication failures, or unsupported endpoints) are returned directly in the response body, alongside the relevant HTTP status code.

Transaction-level errors (such as an issuer decline or 3D Secure failure) are returned within the transaction response body itself. In these cases, the API request is considered valid and successful at the gateway level (typically a 200 response), but the transaction result reflects the decline or failure.

HTTP STATUS CODE SUMMARY
200 OK Everything worked as expected
400 Bad Request The request was unacceptable, often due to missing a required parameter
401 Unauthorized No valid Secret Token provided
403 Forbidden The Secret Token doesn’t have permissions to perform the request
404 Not Found The requested resource doesn't exist
405 Not Allowed The request method is not supported for the requested resource
406 Not Acceptable The requested resource could not be processed due to a conflict with this or related resources
422 Failed Validation The request payload could not pass the validation (probably incorrect)
429 Too Many Requests Too many requests hit the API too quickly. We recommend an exponential backoff of your requests
500 Server Errors Something went wrong on NPGW’s end (these are rare)

One-time payments

  1. When a customer is ready to complete a purchase, your application will submit a Transaction to the NPGW API.
  2. The API provides a URL that redirects customers to an NPGW-hosted page for 3D Secure.
  3. Customers authenticate with their bank when required to complete the transaction.
  4. Asynchronously, the NPGW API delivers notifications related to the Transaction and your application can further poll status.

Core Resources

Token

The Token endpoint is used to create a new Bearer token  or refresh an expired one for accessing the NPGW API. Tokens are valid for 15 minutes but can be regenerated at any time.

The request requires a plain text Secret Token within the body. A successful response returns a JSON object containing the ID Token which can be used for further operation of the NPGW API.

ENDPOINTS
POST /merchant-v1/token See details

Payment

Posting to the Transaction endpoint enables a new credit/debit card payment to be submitted for processing.

The NPGW supports both SALE and AUTH transaction types (the latter requiring a separate Capture action to be submitted to complete a payment). On success, the NPGW API returns a secure Payment URL  to which the end customer must be redirected in order to complete 3D Secure enrollment checks and authentication with their bank where necessary.

The Transaction object

For full detail of the required request structure for and Gateway responses post Transaction processing please review our API Summary. You can find a direct link to specific sections via links in the endpoints tables.

ENDPOINTS
POST /merchant-v1/transaction See details
Heads up!

The optional async query parameter (a boolean) determines whether a transaction should be processed asynchronously with the NPGW - if true, the API responds immediately while processing continues in the background.

Payment Operations

The NPGW API provides further Management  functions that can be used with previously processed Transactions in order to collect or return funds as necessary. The full Transaction response object is returned to confirm the status of each operation.

Capture

This endpoint enables Capture of an AUTH Transaction Type with status Authorised by referencing its transactionId.

The NPGW supports both full and partial captures via this operation.

ENDPOINTS
POST /merchant-v1/transaction/{transactionId}/capture See details

Cancel

This endpoint enables Cancellation (or Void), in full, of an AUTH  Transaction Type with status Authorised by referencing its transactionId. A cancellation  may only be submitted against a Transaction which has no captures associated with it.

Please note that this operation is irreversible. Once this operation has been completed successfully, if you wish to charge the customer again  then a new transaction must be submitted.

ENDPOINTS
POST /merchant-v1/transaction/{transactionId}/cancel See details

Refund

This endpoint enables Refund of a successful Transaction by referencing its transactionId.

The NPGW supports both full and partial refunds via this operation.

ENDPOINTS
POST /merchant-v1/transaction/{transactionId}/refund See details

Refund against an individual capture

This endpoint enables Refund  against a successful Capture within a Transaction by referencing the transactionId and the individual operationId of the capture.

ENDPOINTS
POST /merchant-v1/transaction/{transactionId}/refund/capture/{operationId} See details

Payment Status

Querying the Transaction endpoint allows you to retrieve the full detail and current status of a Transaction using either the NPGW transactionId or your own submitted externalTransactionId.  The full Transaction response object is returned.

ENDPOINTS
GET /merchant-v1/transaction See details

Notifications

The NPGW is able to post notifications to your servers for each event that occurs during the lifecycle of a Transaction.

Notifications will be automatically sent if a valid, fully qualified HTTPS-secured endpoint is provided in the notificationUrl field with each submitted Transaction. The notification request body will contain the full Transaction response object. You can append query string parameters (e.g. a security token) to the notificationUrl. These parameters will be returned unmodified with each notification, allowing you to verify that the communication originated from our gateway.

Heads up!

The NPGW sends a single webhook per event — there are no retries if your server is unavailable or does not respond.

They are also sent asynchronously and so may arrive out of order.

Testing

All features of the NPGW can be tested via our Sandbox!

However, we please ask you don’t use the sandbox to load test your integration as it is a non-performant environment.
